无重复字符的最长子串

发表于 2019-03-13 | 阅读次数:

  为跳出周更的坏习，==，结果上一篇发出后隔天就直接写了这道题，因为使用到了，所以又发现了蛮多的坑，包括之前对map的一些误解，估摸着会专门出一篇对map、HashMap、hash_map以及unordered_map的学习笔记吧，因为还是蛮有用到的。

题目

  给定一个字符串，请你找出其中不含有重复字符的最长子串的长度。
示例 1:
输入: “abcabcbb”
输出: 3
解释: 因为无重复字符的最长子串是 “abc”，所以其长度为 3。
示例 2:
输入: “bbbbb”
输出: 1
解释: 因为无重复字符的最长子串是 “b”，所以其长度为 1。
示例 3:
输入: “pwwkew”
输出: 3
解释: 因为无重复字符的最长子串是 “wke”，所以其长度为 3。
请注意，你的答案必须是 子串 的长度，”pwke” 是一个子序列，不是子串。

难度：中等
分析：这道题其实乍一眼看下下去就不会很难的那种，就是你所设计的程序执行的快慢而已。最直观的就是一边一次遍历这个字符串然后根据拿到的字符构建“不重复且连续”字符串，在获取新字符串中也同时进行对子串的遍历查找，若是不巧碰见了使条件失效的字符，那就打翻 这个子串重新以下一个预定的节点开始构建子串，直至队尾，这便是暴力破解法的想法，但其实归根结底也是找子串的这样一个过程，包括后面较优的算法也是这样，只是确认处理重复字符以及新建子串的方式不同而已，下面详述：

题解

暴力破解法

  先以官方的暴力法为例，官方题解中先考虑到了子串在原字符串中是怎么规划的，定义i为子串的头，j（实际上为j-1）为子串的尾，根据数组的特点j-i即为目前子串的长度。在ij双循环遍历的条件下再利用一个判别函数allUnique来鉴定子串中是否已经含有将要加入的下一个字符，代码思路其实很清晰。
  这个方法首先要在i上遍历一遍字符串的长度也就是O(j−i)，对给定的i，j进行滚动时又是一个O(n-i-1)数量级的复杂度，最后在内层利用判别函数的时候也是一个近似O(n)量级的复杂度，最终算法时间复杂度为O(n3)。空间复杂度，O(min(n,m))，我们需要O(k)的空间来检查子字符串中是否有重复字符，其中k表示 Set 的大小。而 Set的大小取决于字符串n的大小以及字符集/字母m 的大小。详细推导请参见力扣题库官方说明，因为实在不好也在页面上表达出来。

c++代码

  因为已经不是第一次碰见查找了，所以当时第一反应是想到了利用两数之和的“哈希表”map，这里注意了，为什么这段文字看起来那么奇怪呢，因为我只是开玩笑地说出了这句话，c++中map的实现其实是基于红黑树实现的，时间复杂度为O(logn)，而我之前误以为这是基于hash表实现的，这可就误会大了，发完这篇之后我也会相应地纠正之前文章的观点。
  接下来大致说明下我这份代码的构思，首先是没有端标记的，利用多设变量的方式我记录了当前所要构建子串的头i的下一个下标，在构建子串的时候直接利用了map来构建，如果有重复元素则先比较子串长度是否大于了记录的最大长度，否则不更新，但map是要清空的，以便后续操作不出问题，以及最后特殊处理是针对某些特殊情况的，比如单空格构成的串，有在意比较是否有意义的判断（即i大于某值后与已经获取的最长长度比较）。

滑动窗口法

  一开始看到这个标题的时候，我还以为是计网呢，然而确实思想上也有很多类似的地方。其实在暴力破解法中之所以会花费大量的时间，主要是在子串在每次出现重复字符之后，我们需要重新开始子串的构建，这花费了大量的时间。实际上我们只要有办法确定出新的字符在子串的位置，一者被包含的该位置往前的子串的子串的长度是小于该子串的，二者我们有办法确保该位置往后到新字符是我们要找的新的不重复的子串片段，就打破了之前完全重建子串的僵局。
  当确认查找对象为某个具体出现的字符的时候，很容易就想到使用HashSet作为滑动窗口，我们可以用O(1)的时间来完成对字符是否在当前的子字符串中的检查。
  借官方题解的说明，滑动窗口是数组/字符串问题中常用的抽象概念。 窗口通常是在数组/字符串中由开始和结束索引定义的一系列元素的集合，即[i,j)（左闭，右开）。而滑动窗口是可以将两个边界向某一方向“滑动”的窗口。例如，我们将[i,j)向右滑动1个元素，则它将变为[i+1,j+1)（左闭，右开）。而在这个解法的处理中，我们就是通过查找s[j]，判断滑动i还是j的。

  这份在整体的执行上就比较优势了，哪怕是在最糟糕的情况下也就是只有单字符重复出现的子串条件下，该算法达到最大复杂度也就是O(2n)，仍然视作O(n)级别的算法来对待。空间复杂度是与之前的方法相同的，因为同样也是要构建子串的，这里不加赘述。

优化滑动窗口法

  那么说到优化，我们其实又是找之前算法的问题，滑动窗口到底缺漏在哪呢，想一想我们之前是利用set来处理i的变更，因为set是有序的啊，按之前的i一个个往下推就有办法找到已进去set的重复元素的位置了。而我们实际上进行这操作的目的就只是将已进去set的重复元素下一个下标赋给i而已。set只有key当然是做不到的，所以就可以想到利用有value的map是不是可以尝试一下，利用原字符串的每个字符作为key,他们的下标作为value，然后再通过key访问重复元素的value直接得到下标，想起来就很舒服，也就得到了优化的算法。
